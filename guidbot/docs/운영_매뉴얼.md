## 1. 서비스 구성 개요

### 1-1. 서비스 목록

| 포트 | 서비스 | 접근 범위 | 메모리 | 특이사항 |
|------|--------|----------|--------|---------|

## 3. 서버 시작 / 정지

### 3-1. 전체 서비스 시작

```cmd
cd D:\MH\guidbot
venv\Scripts\activate.bat
start.bat
```

또는 탐색기에서 `start.bat` 더블클릭.

**start.bat 실행 순서:**

```
[1] 가상환경 활성화
[2] 포트 충돌 정리 (8501~8505 사용 중인 프로세스 종료)
[3] warmup.py 실행 (AI 모델 사전 로딩, 약 2~5초)
    ※ BM25 디스크 캐시 있으면 ~0.3초로 단축
[4] 앱 동시 시작:
    - 병동 대시보드 (8501) → 백그라운드 창
    - 원무 대시보드 (8503) → 백그라운드 창
    - 벡터DB 관리자 (8505) → 백그라운드 창
    - 관리자 대시보드 (8504) → 백그라운드 창
    - AI 챗봇 (8502) → 이 창 포그라운드
[5] 로그 파일 생성: logs\admin_YYYYMMDD.log
```

> **주의**: start.bat 창을 닫으면 모든 서비스가 함께 종료됩니다.
> 운영 중에는 창을 최소화하여 유지합니다.

### 3-2. 개별 서비스 시작

특정 서비스만 재시작이 필요한 경우:

```cmd
venv\Scripts\activate.bat

REM 병동 대시보드만
streamlit run dashboard_app.py --server.port 8501 --server.address 0.0.0.0

REM AI 챗봇만
streamlit run main.py --server.port 8502 --server.address 0.0.0.0

REM 원무 대시보드만
streamlit run finance_app.py --server.port 8503 --server.address 0.0.0.0

REM 관리자 대시보드만 (localhost 전용)
streamlit run admin_app.py --server.port 8504 --server.address 127.0.0.1

REM 벡터DB 관리자만 (localhost 전용)
streamlit run vector_db_admin.py --server.port 8505 --server.address 127.0.0.1
```

> `.streamlit/config.toml`의 공통 설정이 자동 적용됩니다.
> `--server.fileWatcherType none` 은 config.toml에서 설정되므로 별도 인자 불필요.

### 3-3. 전체 서비스 정지

```cmd
taskkill /IM streamlit.exe /F
taskkill /IM python.exe /F
```

> 강제 종료 후 30초 대기 후 재시작하면 Oracle 연결 풀이 안정적으로 초기화됩니다.

### 3-4. 재시작 (권장)

```cmd
taskkill /IM streamlit.exe /F 2>nul
timeout /t 5 /nobreak
start.bat
```

## 4. 업데이트 배포 (deploy.bat)

소스 코드 업데이트 시 `deploy.bat`를 사용합니다.

### 4-1. deploy.bat 동작

```
1단계: 실행 중인 모든 Streamlit 프로세스 종료
2단계: git pull (최신 코드 가져오기)
3단계: requirements.txt 변경 감지 → 변경 시 자동 pip install
4단계: start.bat 호출 (서비스 재시작)
```

### 4-2. 배포 실행

```cmd
cd D:\MH\guidbot
venv\Scripts\activate.bat
deploy.bat
```

### 4-3. 새 패키지 포함 배포 시 주의사항

`requirements.txt`에 새 패키지가 추가된 경우 (예: `torchvision`) 별도 설치가 필요할 수 있습니다:

```cmd
REM torchvision 등 PyTorch 생태계 패키지는 CPU 인덱스 지정 필요
pip install torchvision --index-url https://download.pytorch.org/whl/cpu
```

### 4-4. 수동 배포 (git 없는 환경)

```cmd
REM 1. 서비스 중지
taskkill /IM streamlit.exe /F 2>nul
timeout /t 3 /nobreak

REM 2. 파일 수동 복사 (수정된 파일만)
REM    ui\finance\ 패키지 전체 교체 주의

REM 3. 패키지 업데이트 (requirements.txt 변경 시에만)
pip install -r requirements.txt

REM 4. 서비스 재시작
start.bat
```

## 6. 로그 관리

### 6-1. 로그 파일 위치

```
D:\MH\guidbot\logs\
├── dashboard_YYYYMMDD.log    병동 대시보드 로그
├── chatbot_YYYYMMDD.log      AI 챗봇 로그
├── finance_YYYYMMDD.log      원무 대시보드 로그
├── vecadmin_YYYYMMDD.log     벡터DB 관리자 로그
└── admin_YYYYMMDD.log        전체 서비스 시작/종료 로그
```

파일명 규칙: `서비스명_YYYYMMDD.log` (일별 자동 생성)

### 6-2. 로그 확인 명령어

```cmd
REM 오늘 오류만 확인
findstr /I "ERROR WARNING" logs\chatbot_20260430.log

REM 실시간 로그 확인
powershell Get-Content logs\chatbot_20260430.log -Wait

REM 지난 7일 오류 통계
powershell "Get-ChildItem logs\*.log | Where-Object {$_.LastWriteTime -gt (Get-Date).AddDays(-7)} | ForEach-Object { Select-String 'ERROR' $_.FullName } | Measure-Object"
```

### 6-3. 주요 로그 패턴

| 패턴 | 의미 | 조치 |
|------|------|------|
| `Oracle connection OK` | DB 연결 정상 | - |
| `CircuitBreaker: OPEN` | DB 연속 3회 실패 → 30초 자동 차단 | Oracle 상태 확인, 30초 후 자동 복구 |
| `CircuitBreaker: CLOSED` | DB 연결 복구됨 | - |
| `BM25 캐시 HIT` | BM25 디스크 캐시 적중 (~0.3초) | - |
| `BM25 캐시 MISS → 재구축` | 신규 구축 또는 캐시 무효화 | 최초 1회 정상 (3~8초 소요) |
| `Cache miss` | 검색 캐시 만료, DB 재조회 (정상) | - |
| `[fast] FAISS 검색: 3건` | 빠른 검색 완료 | - |
| `[standard] Hybrid(BM25+FAISS)` | 표준 검색 완료 | - |
| `GOOGLE_API_KEY rate limit` | API 요청 한도 초과 | API 키 추가 설정 |
| `RAGPipeline 싱글톤 리셋` | 지식베이스 재로드 | - |
| `ORA-` | Oracle 오류 코드 | [13-2](#13-2-대시보드-데이터-없음) 참조 |
| `[Lazy] 탭X 진입, 쿼리 실행` | 탭별 Lazy Loading 동작 (정상) | - |

### 6-4. 로그 정리

로그는 30일 자동 보존됩니다. 수동 정리가 필요한 경우:

```cmd
REM 30일 이상 된 로그 삭제
forfiles /p D:\MH\guidbot\logs /m *.log /d -30 /c "cmd /c del @file"
```

## 7. 지식베이스 업데이트

규정집이 개정되거나 신규 지침이 추가될 때 AI 챗봇 지식베이스를 갱신합니다.

### 7-1. PDF 파일 교체

```
D:\MH\guidbot\docs\regulations\ 폴더에
신규 또는 개정된 규정 PDF를 복사합니다.
(기존 파일과 같은 이름이면 자동으로 교체됩니다)
```

### 7-2. 지식베이스 재구축

```cmd
venv\Scripts\activate.bat
python build_db.py --no-sync
```

표(Table) 구조가 많은 규정집(NEDIS 코드표 등)의 경우:

```cmd
python build_db.py --no-sync --use-markdown
```

특정 부서 규정만 빠르게 업데이트:

```cmd
python build_db.py --no-sync --dept 간호부
python build_db.py --no-sync --dept 원무팀
```

**소요 시간**: PDF 100페이지 기준 약 3~10분
**기존 데이터 보호**: 재구축 전 자동으로 `vector_store_backup\`에 백업됩니다.

> 재구축 완료 후 `vector_store/.bm25_cache_*.pkl`이 자동 갱신됩니다.
> 이전 캐시는 문서 변경 감지 시 자동 무효화됩니다.

### 7-3. 재구축 후 챗봇 재시작

지식베이스 재구축 후 챗봇을 재시작해야 새 내용이 반영됩니다.

```cmd
taskkill /IM streamlit.exe /F 2>nul
timeout /t 3 /nobreak
start.bat
```

또는 벡터DB 관리자(8505)에서 **"벡터DB 재로드"** 버튼 클릭 (서비스 중단 없이 반영).

### 7-4. 업데이트 확인

AI 챗봇(http://192.1.1.231:8502)에서 개정 규정 관련 질문으로 응답 확인.

## 8. 관리자 대시보드 (포트 8504)

관리자 대시보드는 **서버 PC에서만** 접근 가능합니다 (`localhost:8504`).

### 8-1. 접속 방법

## 10. 백업 및 복구

### 10-1. 자동 백업 항목

| 항목 | 위치 | 주기 | 보존 수 |
|------|------|------|--------|
| 지식베이스 | `vector_store_backup\` | 재구축 시 자동 | 최대 5개 |
| BM25 캐시 | `vector_store\.bm25_cache_*.pkl` | 재구축 시 자동 갱신 | 1개 (현재) |
| 로그 | `logs\` | 매일 자동 생성 | 30일 |

### 10-2. 수동 백업 (주 1회 권장)

```cmd
REM .env와 벡터DB를 외부 드라이브로 백업
set DT=%date:~0,4%%date:~5,2%%date:~8,2%
xcopy /E /I D:\MH\guidbot\vector_store  E:\backup\guidbot\vector_store_%DT%\
copy D:\MH\guidbot\.env  E:\backup\guidbot\.env.%DT%
```

### 10-3. 지식베이스 복구

벡터DB가 손상된 경우 백업에서 복구합니다.

```cmd
REM 1. 손상된 벡터DB 이동
rename D:\MH\guidbot\vector_store vector_store_broken

REM 2. 백업에서 복구 (폴더명은 실제 백업 폴더명으로 변경)
xcopy /E /I "D:\MH\guidbot\vector_store_backup\20260420_backup" D:\MH\guidbot\vector_store\

REM 3. BM25 캐시 삭제 (복구 후 자동 재생성됨)
del D:\MH\guidbot\vector_store\.bm25_cache_*.pkl 2>nul

REM 4. 서비스 재시작
taskkill /IM streamlit.exe /F
timeout /t 3 /nobreak
start.bat
```

### 10-4. .env 분실 시

```
1. .env.example을 복사하여 .env 재생성
```

### 11-2. RAG 검색 파라미터 조정

`.env`에서 조정 (변경 후 서비스 재시작):

| 설정 | 기본값 | 설명 |
|------|--------|------|
| `RETRIEVE_TOP_K` | 20 | 검색 1차 후보 수 (높을수록 정확하나 느림) |
| `RERANK_TOP_N` | 3 | Cross-Encoder 후 최종 참조 문서 수 |

| `CHUNK_SIZE` | 1000 | PDF 분할 크기 (변경 후 지식베이스 재구축 필요) |

### 11-3. Oracle 연결 풀 크기 조정

동시 접속자가 많아 DB 오류 발생 시:

```ini
# .env
ORACLE_POOL_MAX=15    # 기본값 10, 최대 30
ORACLE_POOL_MIN=3     # 기본값 2
```

### 11-4. Streamlit 서버 설정 변경

`.streamlit/config.toml` 수정 후 서비스 재시작:

```toml
[server]
# fileWatcherType = "none"  ← 절대 삭제하지 않음 (WARNING 폭발 방지)
maxUploadSize = 200         # PDF 업로드 크기 제한 (MB)
maxMessageSize = 100        # WebSocket 메시지 크기 (MB)
```

> **주의**: `fileWatcherType = "none"` 항목은 삭제하지 않습니다.
> 삭제 시 서비스 기동 중 `ModuleNotFoundError: torchvision` WARNING이 수십 회 출력됩니다.

### 11-5. 포트 변경

변경 시 방화벽 규칙도 함께 수정해야 합니다.

## 13. 장애 대응

### 13-1. 서비스 접속 불가

```
증상: 브라우저 "사이트에 연결할 수 없음"
```

1. 프로세스 확인: `tasklist | findstr streamlit`
2. 프로세스 없음 → 서비스 재시작: `start.bat`
3. 프로세스 있음 → 포트 확인: `netstat -ano | findstr 8501`
4. 방화벽 확인: Windows Defender 방화벽 > 인바운드 규칙

### 13-2. 대시보드 데이터 없음

```
증상: 대시보드는 열리지만 수치가 모두 0 또는 "-" (Oracle 연결 실패)
```

1. Oracle 서버 네트워크 확인:
   ```cmd
   ping 192.168.x.x
   ```
2. Oracle 서비스 상태 확인 (DBA에게 요청)
3. 계정 잠금 여부 확인:
   ```sql
   SELECT ACCOUNT_STATUS FROM DBA_USERS WHERE USERNAME='GUIDBOT_READONLY';
   ```
4. `.env`의 `ORACLE_PASSWORD` 재확인
5. CircuitBreaker 상태 확인 — 로그에서 `Circuit breaker OPEN` 검색:
   ```cmd
   findstr "Circuit breaker" logs\dashboard_20260430.log
   ```
   → OPEN 상태이면 30초 후 자동 복구됩니다.

### 13-3. AI 챗봇 답변 없음

```
증상: 질문 입력 후 무한 로딩 또는 오류 메시지
```

1. API 키 확인:
   ```cmd
   python -c "import os; from dotenv import load_dotenv; load_dotenv(); k=os.getenv('GOOGLE_API_KEY',''); print('KEY:', k[:10]+'...' if k else '없음')"
   ```
2. API 사용량 한도 확인: https://aistudio.google.com
3. 지식베이스 파일 확인:
   ```cmd
   dir D:\MH\guidbot\vector_store\
   ```
   `index.faiss`와 `index.pkl` 두 파일이 있어야 합니다.

### 13-4. 메모리 부족으로 자동 종료

```
증상: 서비스 갑자기 종료, 이벤트 뷰어에 OOM 기록
```

1. 작업 관리자에서 메모리 사용량 확인
2. **단기 조치**: AI 챗봇(8502)을 일시 중지하고 대시보드만 운영
3. **영구 조치**: RAM 16GB 이상 확보 (AI 챗봇 ~2GB 필요)

### 13-5. Circuit Breaker 발생

```
증상: 로그에 "Circuit breaker OPEN", 병동 대시보드 데이터 공백
```

Oracle DB가 일시 응답 불가로 3회 연속 실패한 상황입니다.
**30초 후 자동 복구**됩니다. 30초 경과 후에도 지속 장애 시 DBA에게 문의합니다.

```
Circuit Breaker 동작 규칙:
  - 3회 연속 실패 → OPEN (30초 차단)
  - 30초 경과 → HALF-OPEN (테스트 쿼리 1회)
  - 성공 → CLOSED (정상 운영 복귀)
  - 실패 → 다시 OPEN (30초 재차단)
```

### 13-6. Streamlit WARNING 폭발 (torchvision)

```
증상: 서비스 기동 시 콘솔에 "ModuleNotFoundError: No module named 'torchvision'"
      WARNING이 수십 줄 출력됨
```

원인: `transformers 5.x` 이미지 프로세서의 lazy import + Streamlit 파일 와처

**즉각 조치 (2단계):**

```cmd
REM 1. torchvision 설치
pip install torchvision --index-url https://download.pytorch.org/whl/cpu

REM 2. .streamlit/config.toml 확인
type .streamlit\config.toml | findstr "fileWatcherType"
REM 출력: fileWatcherType = "none"  ← 이 줄이 있어야 함
```

없으면 `.streamlit/config.toml`에 `fileWatcherType = "none"` 추가 후 재시작.

### 13-7. 원무 대시보드 탭 오류

```
증상: 특정 탭(지역별 분석 등) 진입 시 오류 또는 빈 화면
```

`ui/finance/` 패키지의 개별 탭 모듈 확인:

```cmd
REM 지역 탭 오류 시 folium 확인
python -c "import folium; from streamlit_folium import st_folium; print('folium OK')"

REM 모듈 오류 시 재설치
pip install folium streamlit-folium --upgrade
```

### 13-8. BM25 검색 결과 이상

```
증상: 검색 결과 품질 저하, 관련 없는 문서 반환
```

BM25 캐시가 오래된 인덱스를 참조할 수 있습니다:

```cmd
REM BM25 캐시 삭제 (재구축 시 자동 재생성)
del D:\MH\guidbot\vector_store\.bm25_cache_*.pkl
```

서비스 재시작 시 BM25 인덱스를 재구축하고 새 캐시를 저장합니다 (최초 1회 3~8초 소요).

### 13-9. 긴급 재설치

모든 조치에도 복구 불가 시:
