## 1. 서비스 구성 개요

### 1-1. 구성도

```
[내부망 PC/태블릿]
        |
        | HTTP (내부망)
        |
[가이드봇 서버 Windows]
        |
        +-- 병동 대시보드   :8501  (dashboard_app.py)
        |     - 실시간 병상 현황 / 진료과별 재원 / 수술 통계
        |
        +-- AI 챗봇        :8502  (main.py)
        |     - 규정집 RAG 검색 / 자연어 질의응답
        |
        +-- 재무 대시보드   :8503  (finance_app.py)
              - 수납 현황 / 보험별 매출 / 미수 통계
        |
        | Oracle JDBC (내부망)
        |
[Oracle DB 서버]
```

### 1-2. 서비스별 역할

| 서비스 | 포트 | 주요 사용자 | 특이사항 |
|--------|------|------------|---------|
| 병동 대시보드 | 8501 | 간호사, 원무팀, 병동관리 | 5분 캐시, Oracle 실시간 조회 |
| AI 챗봇 | 8502 | 전직원 | AI 모델 상주 (메모리 2GB) |
| 재무 대시보드 | 8503 | 원무팀, 재무팀 | 5분 캐시, Oracle 실시간 조회 |

### 1-3. 설치 경로 구조

```
D:\guidbot\
├── .env                  ← 설정 파일 (API 키, DB 연결 정보)
├── start.bat             ← 서비스 시작 스크립트
├── vector_store\         ← AI 지식베이스 (FAISS 인덱스)
├── vector_store_backup\  ← 자동 백업 (최대 5개 버전)
├── docs\                 ← 규정집 PDF 및 매뉴얼
├── logs\                 ← 일별 로그 파일
└── data_cache\           ← AI 모델 캐시 (최초 1회 다운로드)
```

## 3. 서버 시작 / 정지

### 3-1. 정상 시작

```cmd
cd D:\guidbot
venv\Scripts\activate.bat
start.bat
```

또는 탐색기에서 `start.bat` 더블클릭.

**시작 순서:**
1. 기존 프로세스 종료 (포트 정리)
2. 가상환경 활성화
3. AI 모델 사전 로딩 (warmup, 약 5~10초)
4. 병동 대시보드 시작 (8501)
5. 재무 대시보드 시작 (8503)
6. AI 챗봇 시작 (8502)

### 3-2. 개별 서비스 시작

특정 서비스만 재시작이 필요한 경우:

```cmd
venv\Scripts\activate.bat

# 병동 대시보드만
streamlit run dashboard_app.py --server.port 8501 --server.address 0.0.0.0 --server.headless true

# AI 챗봇만
streamlit run main.py --server.port 8502 --server.address 0.0.0.0 --server.headless true

# 재무 대시보드만
streamlit run finance_app.py --server.port 8503 --server.address 0.0.0.0 --server.headless true
```

### 3-3. 정상 정지

```cmd
# 모든 Streamlit 프로세스 종료
taskkill /IM streamlit.exe /F
taskkill /IM python.exe /F
```

> **주의**: 강제 종료는 Oracle 연결 풀이 정리되지 않을 수 있습니다.  
> 가능하면 30초 대기 후 재시작하세요.

### 3-4. 재시작 (권장 방법)

```cmd
taskkill /IM streamlit.exe /F 2>nul
timeout /t 5 /nobreak
start.bat
```

## 4. 상태 모니터링

### 4-1. 헬스체크 URL

각 서비스의 상태를 JSON으로 확인합니다:

```
http://서버IP:8501/?health=1
http://서버IP:8502/?health=1
http://서버IP:8503/?health=1
```

응답 예시:

```json
{
  "status": "ok",
  "ts": 1745000000.0,
  "oracle": true,
  "oracle_msg": "Connection OK",
  "memory_rss_mb": 512.3,
  "system_memory_pct": 45.2,
  "system_memory_avail_gb": 8.5
}
```

| 항목 | 정상 | 주의 | 위험 |
|------|------|------|------|
| `status` | `"ok"` | - | `"error"` |
| `oracle` | `true` | - | `false` |
| `memory_rss_mb` | < 1500 | 1500~2500 | > 2500 |
| `system_memory_pct` | < 70% | 70~85% | > 85% |

### 4-2. 프로세스 확인

```cmd
tasklist | findstr streamlit
tasklist | findstr python
```

정상 시 출력 예:

```
streamlit.exe    1234  ...  200,000K
streamlit.exe    2345  ...  200,000K
streamlit.exe    3456  ...  2,000,000K  ← AI 챗봇 (메모리 높음이 정상)
```

### 4-3. 포트 사용 확인

```cmd
netstat -ano | findstr "8501\|8502\|8503"
```

## 5. 로그 관리

### 5-1. 로그 파일 위치

```
D:\guidbot\logs\
├── dashboard_20260417.log   ← 병동 대시보드 로그
├── finance_20260417.log     ← 재무 대시보드 로그
└── chatbot_20260417.log     ← AI 챗봇 로그
```

파일명 규칙: `서비스명_YYYYMMDD.log` (일별 자동 생성, 30일 보존 후 자동 삭제)

### 5-2. 로그 확인

```cmd
# 오늘 오류만 확인
findstr /I "ERROR WARNING" logs\dashboard_20260417.log

# 실시간 로그 확인
powershell Get-Content logs\chatbot_20260417.log -Wait
```

### 5-3. 주요 로그 패턴

| 패턴 | 의미 | 조치 |
|------|------|------|
| `Oracle connection OK` | DB 연결 정상 | - |
| `Circuit breaker OPEN` | DB 연속 3회 실패, 30초 차단 | Oracle 상태 확인 |
| `Cache miss` | 캐시 만료, DB 재조회 | 정상 (5분 주기) |
| `GOOGLE_API_KEY rate limit` | API 요청 한도 초과 | API 키 추가 설정 |
| `ORA-` | Oracle 오류 코드 | [11. 장애 대응](#11-장애-대응) |

```cmd
taskkill /IM streamlit.exe /F 2>nul
```

## 8. 백업 및 복구

### 8-1. 자동 백업 항목

| 항목 | 위치 | 주기 | 보존 수 |
|------|------|------|--------|
| 지식베이스 | `vector_store_backup\` | 재구축 시 자동 | 최대 5개 |
| 로그 | `logs\` | 매일 자동 생성 | 30일 |

### 8-2. 수동 백업 (권장: 주 1회)

```cmd
# .env와 vector_store를 외부 드라이브로 백업
xcopy /E /I D:\guidbot\vector_store  E:\backup\guidbot\vector_store_%date:~0,4%%date:~5,2%%date:~8,2%\
copy D:\guidbot\.env  E:\backup\guidbot\.env.%date:~0,4%%date:~5,2%%date:~8,2%
```

### 8-3. 지식베이스 복구

```cmd
# 백업에서 복구
xcopy /E /I D:\guidbot\vector_store_backup\[백업폴더명] D:\guidbot\vector_store\
```

이후 서비스 재시작:

```cmd
taskkill /IM streamlit.exe /F
timeout /t 3 /nobreak
start.bat
```

### 8-4. .env 분실 시

1. `.env.example`을 복사하여 `.env` 재생성
2. Oracle DB 담당자에게 DB 계정 비밀번호 재확인
3. Google API 키: https://aistudio.google.com 에서 재발급
4. 관리자 비밀번호 새로 설정 후 전 직원 공지

## 10. 주요 설정 변경

### 10-1. API 키 교체/추가

`.env` 파일 수정:

```ini
GOOGLE_API_KEY="새로운키"
GOOGLE_API_KEY_2="추가키2"
```

변경 후 서비스 재시작 필요.

### 10-2. AI 응답 품질 조정

`.env` 파일에서 조정:

| 설정 | 기본값 | 설명 |
|------|--------|------|
| `RETRIEVE_TOP_K` | 20 | 검색 후보 수 (높을수록 정확, 느림) |
| `RERANK_TOP_N` | 3 | 최종 참조 문서 수 |
| `LLM_TEMPERATURE` | 0.1 | 답변 일관성 (0=일관, 2=창의적) |
| `CHUNK_SIZE` | 1000 | PDF 분할 크기 (변경 후 재구축 필요) |

### 10-3. Oracle 연결 풀 크기 조정

동시 접속자가 많아 DB 오류가 발생하는 경우:

```ini
# .env
ORACLE_POOL_MAX=15    # 기본값 10, 최대 30
ORACLE_POOL_MIN=3     # 기본값 2
```

### 10-4. 포트 변경

`.env` 수정 후 방화벽 규칙도 함께 변경합니다:

```ini
# 포트는 .env가 아닌 start.bat에서 변경
```

`start.bat`에서 `--server.port` 인수를 수정합니다.

## 11. 장애 대응

### 11-1. 서비스 접속 불가

```
증상: 브라우저에서 "사이트에 연결할 수 없음"
```

1. 프로세스 확인: `tasklist | findstr streamlit`
2. 프로세스 없으면 → 서비스 재시작: `start.bat`
3. 프로세스 있으면 → 포트 확인: `netstat -ano | findstr 8501`
4. 방화벽 확인: Windows Defender 방화벽 > 인바운드 규칙

### 11-2. 대시보드 데이터 없음 ("Oracle 연결 실패")

```
증상: 대시보드는 열리지만 모든 수치가 0 또는 "-"
```

1. Oracle 서버 네트워크 연결 확인:
   ```cmd
   ping 192.168.x.x    ← Oracle 서버 IP
   ```
2. Oracle 서비스 상태 확인 (DBA에게 요청)
3. 계정 잠금 여부 확인:
   ```sql
   SELECT ACCOUNT_STATUS FROM DBA_USERS WHERE USERNAME='GUIDBOT_READONLY';
   ```
4. `.env`의 비밀번호 재확인

### 11-3. AI 챗봇 답변 없음

```
증상: 질문 입력 후 무한 로딩 또는 오류 메시지
```

1. API 키 확인:
   ```cmd
   python -c "import google.genai, os; from dotenv import load_dotenv; load_dotenv(); print('KEY:', os.getenv('GOOGLE_API_KEY')[:10] + '...')"
   ```
2. API 한도 초과 여부 확인: https://aistudio.google.com 에서 사용량 확인
3. 지식베이스 파일 확인:
   ```cmd
   dir D:\guidbot\vector_store\
   ```
   `index.faiss`와 `index.pkl` 파일이 있어야 합니다.

### 11-4. 메모리 부족으로 서비스 자동 종료

```
증상: 서비스가 갑자기 종료됨, 이벤트 로그에 OOM 기록
```

1. 작업 관리자에서 메모리 사용량 확인
2. 단기 조치: AI 챗봇(8502)을 일시 중지하고 대시보드만 운영
3. 영구 조치: RAM 16GB 이상으로 증설 (AI 챗봇은 약 2GB 사용)

### 11-5. Circuit Breaker (DB 차단) 발생

```
증상: 로그에 "Circuit breaker OPEN" 메시지, 30초간 DB 조회 차단
```

Oracle DB가 일시적으로 응답하지 않아 3회 연속 실패한 상황입니다.  
30초 후 자동 복구됩니다. DB 지속 장애라면 DBA에게 문의합니다.

### 11-6. 긴급 재설치

모든 조치에도 복구가 안 될 경우:

```cmd
# 1. 기존 가상환경 삭제
rmdir /S /Q venv

# 2. 가상환경 재생성
python -m venv venv
venv\Scripts\activate.bat

# 3. 패키지 재설치
pip install -r requirements.txt

# 4. 서비스 시작
start.bat
```

> `.env`, `vector_store\`, `docs\` 폴더는 삭제하지 않습니다.
